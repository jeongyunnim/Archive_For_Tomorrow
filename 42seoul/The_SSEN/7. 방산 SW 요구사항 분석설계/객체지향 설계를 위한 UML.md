## UML Notation의 이해
### class 다이어그램
- 퍼블릭을 기준으로, 꼭 알아야 하는 private을 적는다.
- 스테레오 타입
	- interface, utility, abstract, enumeration
	- `<<>>`로 감싸서, 이탤릭체로
- [ ] 라이브러리로 빼라. 웹서브 꼭 재사용하고 싶다면...
- 의존, 연관, 상속, 집합 관계, 합성 관계, 구현관계
	- 실선으로 표시하면 클래스 상속
	- 점선으로 표시하면 인터페이스를 구현한 것.
### Use Case
- extend:  옵션, 해도되고 안 해도 되고
	- 포인트 결제
	- `<-`
- [ ] include와 extend의 차이
### state machine 다이어그램
- 상태와 모드
### activity 다이어그램
- 시퀀스 다이어그램과 뭐가 다른가?
	- 각 활동이 있다.
- 병렬 처리를 표현할 수 있다.
![[Excalidraw/LIG Nex1 방산 소프트웨어.md#^group=5fZYSRwJd280xXp4ZRAs-]]
- 스윕레인
### 시퀀스 다이어그램
- 요구분석을 가지고 시나리오를 그릴 때 무조건 그리게 된다.

- 그리는 이유
	- Use caseㄹ를 실현하기 위해
- turnkey

	- "Object Name : Class Name"으로 표기

- 디자인을 하면 코드의 반절은 완성한 것이다.

	- 클래스 추출

### 아키텍처 설계

### 요구사항 분석: 요구사항 파악
- 도출방법
	- 최종 사용자가 현 시스템의 문제점 및 개발 시스템에 대한 요구사항 작성
	- 시스템 분석가와 면담, 설문, 기존 시스템 분석을 통해 작성
- 필요사항
	- 개발 시스템 업무 영역이 명확히 정의
	- 사용자의 검증
- 노테이션의 내용은 크게 중요하지 않다.
	- 이런 것들을 알아야,, 현업에서 무조건 써야하기 때문에
- UML를 통해서 OOAD를 
	- 분석도하고 설계도 하고 UML이
- [ ] [[Head First Object Oriented Analysis And Design]] 별도로 해야 한다.

- Usecase disgram
	- 모든 액터는 하나 이상의 Use Case
	- 시험통제기가 시나리오 배포
- Actor를 찾는 방법
	- 주요 기능을 사용할 사람은 누구인가?
	- 개발 시스템의 지원을 필요로 하는 이
	- 개발 시스템의 유지하고 관리해야 할 필요가 있는 사람
	- 하드웨어 장치
	- ...
- -> Use case 명세서
	- 서술형으로 기술 -> 시나리오 기술
- Use case 실현
	- 어떤 어떤 객체들이 어떻게 상호작용 해야 하는가
	- 상호작용하는 객체를 찾아내고 상호작용을 기술
	- 시나리오를 기술하는 것
- **기본 설계와 상세설계가 있다.**
	- 기본 설계: 다이어그램이 나와야 함 -> 주요 트리거들이 나와야 한다.
	- 상세 설계: 어떤 언어, 프레임워크, 환경에서 할 거냐

### 클래스 설계
- 클래스
	- 어느 컨텍스트에서 쓸 거다. 라는 모델링이 된 상태
	- 추상화
	- OCP
- 관계성
	- 클래스 간의 관계
	- 연관, 포함, 상속, 종속
- state machine diagram
	- 클래스로 생성되는 객체의 모델링
	- was가 뻗었다. 실패 상태를 정의 해서 주변 클래스가 어떤 일을 해야 하는지 정의
### 컴포넌트 설계
- 논리 및 물리 아키텍처
	- 논리: 이미 이전 단계에서 나온 애들 고도
	- 물리: 컴포넌트, 디플로이 다이어그램
### 컴포넌트 배치
- 컴포넌트와 인터페이스
- 프로세스 모델링
	- 프로세스와 스레드 정의
- 디플로이
### 객체지향 분석 및 설계의 최종 목표
- 고객의 요구사항을 모두 만족시키는 것
- 구현자가 문제 없이 구현 가능하도록
### 객체지향 분석 및 설계
- 객체를 찾아내고, 객체의 연관관계를 찾는다.
	- 시나리오를 보고 모델을 찾아내고
	- 모델을 보고 시퀀스 다이어그램
### 설계의 세부 과정
- 클래스 식별
- 클래스 간 상호작용 식별
- 클래스 검증

-  디자인 패턴
```

->고객 요구사항 정리 
-> 도메인 분석을 통한 시스템의 특징 정리
-> 설계도 작성(Use Case 다이어그램)
	다이어그램, 시나리오, 시퀀스 다이어그램
-> 아키텍처 설계(품질 속성)
	디자인 패턴
-> 큰 문제를 작은 문제로 분할 하여 설계(클래스와 연관관계)
-> 작은 문제 해결을 위한 디자인 패턴
```
